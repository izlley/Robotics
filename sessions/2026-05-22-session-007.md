## 2. 한 일

### A. §3.3 (b) — Tiling 제거 + Pixel shuffle 상세 (~100줄 추가)

**Tiling 설명**:
- 고해상도 이미지를 격자로 잘라 각 타일을 vision encoder에 독립 통과
- LLaVA-NeXT, InternVL, SmolVLM-2 기본 설정
- 토큰 수 5배 폭발 → quadratic cost ~25배
- SmolVLA: 제거. global 이미지(512×512) 한 장만 사용
- Robotics에서 OK한 이유: fixed view + multi-camera로 multi-view 보강

**Pixel shuffle 설명**:
- Shi 2016 super-resolution 원조 기법 (sub-pixel convolution)
- VLM에서는 inverse 방향 (downsampling)
- 공간 2×2 block 4 patch를 channel로 옮김: [N, d] → [N/4, 4d] → linear → [N/4, d]
- PyTorch one-liner: `F.pixel_unshuffle(features_spatial, downscale_factor=2)`
- Average pooling 대비 정보 손실 없음 (linear projection이 4 patch 정보 다 봄)
- Token Merging (ToMe) 사촌격 — 공간적 grouping 기반

**누적 효과 표**:
| 단계 | tokens/frame | 비용 비율 |
|---|---|---|
| SmolVLM-2 raw | 1280 | 1× |
| Tiling 제거 | 256 | 1/25 |
| + Pixel shuffle | 64 | 1/400 |

이게 SmolVLA의 "6 Hz on 4090" 효율의 핵심 기술적 이유.

### B. §5.1 Loss — Prediction vs Target 명확화 (~80줄 보강)

**문제**: 사용자가 "prediction은 어디 있냐"고 질문. Flow matching paradigm을 처음 만나는 사람에게는 매우 자연스러운 혼동.

**보강 내용**:
1. **Direct regression(ACT) vs Flow matching(SmolVLA, π0) 비교표**:
   - Direct: model 출력 = action, loss = ||pred_action - GT_action||²
   - Flow: model 출력 = **velocity**, loss = ||pred_velocity - target_velocity||²

2. **수식 시각적 해부** (`\underbrace`로 prediction/target 명시)

3. **Target velocity 어떻게 나오는지**:
   - $A_t^\tau = \tau A_t + (1-\tau)\epsilon$의 미분 = $A_t - \epsilon$
   - 부호 뒤집은 $\epsilon - A_t$가 논문 표기 (제곱이라 부호 무관)
   - 사람이 만든 supervision label이 아니라 **자동 derive**

4. **학습 step 한 번의 의사코드** — 사용자가 머릿속에서 follow 가능

5. **Inference 시점에 action이 어디서 나오는지** — ODE 적분 10 step

6. **Velocity parameterization의 장점 (5가지)** — multimodal, smooth, stable, continuous, few-step

## 4. 학습한 것

### Flow matching paradigm의 핵심 명확화
- "Model이 action을 예측"이 아니라 "velocity field를 예측"
- 사용자가 expect한 "pred - GT" 구조는 velocity 공간에서 그대로 성립
- 학습 - inference 비대칭: 학습은 velocity, inference는 ODE 적분으로 action 복원

### Pixel shuffle의 일반 원리
- "공간 ↔ 채널 정보 이동" — 정보 보존 + 채널 reshape
- Super-resolution(공간↑채널↓)과 vision token 압축(공간↓채널↑)이 같은 연산의 두 방향
- PyTorch: `pixel_shuffle`(up) / `pixel_unshuffle`(down)
- 평균 pooling 대비 정보 손실 없음

### 학습 자료(.md)와 학습자 간 interactive loop의 가치
- self-contained 작성 노력에도 사용자 질문이 모호점 노출
- 즉시 보강 + commit으로 "재방문하면 더 나은 자료"라는 자기개선 ratchet
